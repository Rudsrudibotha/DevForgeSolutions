================================================================================
1. HIGH-LEVEL OVERVIEW
================================================================================
Kinder Care Hub is a multi-tenant SaaS pre-school management system.
It runs as a single Node.js / Express process, connects to Azure SQL
Database for persistence, and uses a private on-disk folder
(`storage/message-attachments/`) for image attachments. The front-end
is a mix of:
   - server-rendered EJS pages (the new "SSR portal") for the
     DevForge, SMS, and Parent dashboards
   - the legacy static SPA in public/ (still mounted at /devforge.html,
     /parent.html, /school/index.html) for backwards compatibility

The architectural pattern is strict three-layer separation:
   HTTP -> Route (application) -> Service (business) -> Repository (data)
Plus a few orthogonal modules:
   - middleware/  : rate limiting, auth, audit, CSRF, scope, sessions
   - security/    : OAuth providers, JWT, school permissions, test auth
   - observability/: Application Insights setup
   - business/sessionContextService.js : canonical ActiveTenantId
                                          session enrichment

The canonical authorisation model is the nine-step access-check order
defined in Task 13 of the SaaS task list. Every protected action runs
through it. The order is:
   1. Authenticate user             (authenticateToken)
   2. Validate active tenant        (sessionContext.IsSchoolUser, etc.)
   3. Check active subscription      (canTenantUseFeature)
   4. Check feature entitlement     (canTenantUseFeature)
   5. Check feature limit            (canTenantUseFeatureWithinLimit)
   6. Check user role permission     (requireSchoolPermission)
   7. Check record belongs to tenant (service-level tenant filter)
   8. Check conversation membership (canUserAccessConversation, etc.)
   9. Execute action + write audit

2. REQUEST LIFECYCLE
================================================================================
Incoming HTTP request -> src/app.js
  1. helmet() applies a strict Content-Security-Policy:
     default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline';
     frame-ancestors 'none'. inline styles are allowed for htmx+alpine.
  2. cors() permits only ALLOWED_ORIGINS (env-configured).
  3. Per-IP rate limiter at /api/* (1500 / 15 min by default).
  4. Tighter per-route rate limiter at /api/users/login and
     /api/users/register (20 / 15 min by default).
  5. express.json({ limit: '256kb' }) parses JSON bodies. The OFX
     upload uses multer with its own 5 MB limit. The image upload
     route uses multer with 10 MB limit.
  6. Static files are served from public/ with aggressive caching.
  7. The new portal router is mounted at '/' (see below). All other
     routers are mounted at /api/*.
  8. 404 handler renders errors/404.ejs.
  9. Global error handler logs a sanitized error (Authorization /
     Cookie / OAuth code stripped).

Portal router (src/application/portal/index.js):
   - loadUser           : reads JWT (header OR kch_token cookie) and
                          populates req.user
   - issueCsrf + verifyCsrf : sets kch_csrf cookie + hx-headers
   - /auth/*            : authRoutes (login, register, OAuth callbacks)
   - /account           : requireAuth (any role) + accountRoutes
   - /sms/*             : requireAuth + requireRole('school','admin')
                          + loadCurrentSchool + scopeToSchool + smsRoutes
   - /parent/*          : requireAuth + requireRole('parent') + parentRoutes
   - /devforge/*        : requireAuth + requireRole('admin') + devforgeRoutes

18a. PARENT VERIFICATION (NEW)
================================
Browser
  -> GET /parent-verify
  -> POST /api/parent-verification/start { email, cellphone }
     parentVerificationRoutes -> parentVerificationService.startVerification
       - validate email + cellphone format
       - check eligibility via DB (Users OR ParentRegistrationRequests)
       - generate emailToken (32 bytes base64url) and smsCode (6 digits)
       - hash both and insert ParentVerificationChallenges row
       - send email with verify link to email
       - (in prod) send SMS with smsCode
       - (in dev) return smsCode in the response so dev can complete
  -> User clicks the link in email:
     GET /parent-verify?token=&email=
  -> POST /api/parent-verification/complete-email { token, email }
     service.completeEmail: hash token, find challenge, mark
     EmailVerified=1, generate + send SMS code
  -> POST /api/parent-verification/complete-sms { token, email, smsCode }
     service.completeSms: hash smsCode, find challenge with
     EmailVerified=1, SmsCodeHash match, mark SmsVerified=1, ensure
     parent user (create if missing), issue 24h magic link, send email
  -> User clicks magic link in email:
     GET /api/parent-verification/magic?token=&email=
     service.consumeMagicLink: hash token, find ParentMagicLinks row,
     mark UsedAt, issue real JWT, return mustSetPassword: true
  -> Browser sets kch_token cookie automatically
  -> Redirect to /parent/portal and prompt to set password

18b. BANK RECONCILIATION (TASKS 1-15)
================================
Finance sidebar (navSpec.js):
  - Import Bank Statement  -> GET /sms/bank-statements/import
  - Reconciliation         -> GET /sms/bank-reconciliation

Legacy CSV upload (/sms/bank-statements/*) remains for backwards
compatibility but is no longer in the Finance nav.

School isolation (mandatory):
  - Every statement/import/transaction row carries TenantId + SchoolId.
  - Portal routes resolve tenant from Schools.TenantId via schoolDb.
  - assertBankAccountForSchool() blocks cross-school account access.
  - matchBankTransactionToInvoice() uses approveBankMatch(), which
    rejects invoices where invoice.SchoolID !== payment.SchoolID.
  - AI reconciliation context (aiContextBuilder) filters invoices by
    ActiveSchoolId only.
  - New tables registered in scopedDb.js SCOPED_TABLES.

Browser
  -> GET /sms/bank-statements/import
     -> Renders import.ejs with school-scoped bank accounts list
  -> POST /api/bank-reconciliation/imports (multipart, file=ofxFile)
     sms-bank-reconciliation-routes.js:
       - verify bank account belongs to ActiveSchoolId
     bankReconciliationService.importOFX:
       - compute SHA256 of OFX file content
       - check duplicate (TenantId+SchoolId+BankAccountId+FileHash+Year+Month)
       - get-or-create BankReconciliationStatement for the month
       - insert BankStatementImport summary row (status='Processing')
       - parse OFX (parseOFX): BankEffectiveDate = PostedDate || TransactionDate
       - for each tx: inMonth() filter, dedup via FITID or hash, insert
         BankTransaction; create one Payment (Transactions row with
         BankTransactionId) per credit via transactionRepository
       - update import summary (counts)
  -> GET /sms/bank-reconciliation
     -> Renders reconciliation-landing.ejs (bank account picker)
  -> GET /sms/bank-reconciliation?bankAccountId=X&filter=open|reconciled|all
     -> Renders reconciliation-list.ejs (monthly statements for that account)
        Open filter = Status IN ('Open','InProgress')
  -> GET /sms/bank-reconciliation/:id?tx=N
     -> getStatement(id, tenantId, schoolId) — 404 if wrong school
     -> Renders reconciliation-detail.ejs (transactions + manual match form)
  -> POST /sms/bank-reconciliation/:id/transactions/:txId/match
     bankReconciliationService.matchBankTransactionToInvoice:
       - verify statement + bank tx belong to tenant + school
       - approveBankMatch(payment.TransactionID, invoiceId) — same-school only
       - set BankTransaction.Status = 'Matched'
  -> User clicks "Suggest match" -> POST /api/ai/reconcile
     aiChatService.suggestBankReconciliationMatch (school-scoped context)
  -> POST /sms/bank-reconciliation/:id/reconcile
     bankReconciliationService.markReconciled:
       - assert statement belongs to tenant + school
       - block only if any transaction still has Status = 'Imported'
       - set status=Reconciled, ReconciledByUserId, ReconciledAt

Browser

18d. AUTO-INVOICING (BACKGROUND)
================================
Scheduler (src/app.js startMonthlyInvoiceScheduler):
  -> every 1st of the month at 00:05, OR immediately on cold start
     if today is the 1st
  -> invoiceService.generateMonthlyInvoices({ Role: 'admin' })
     for each active school:
       - load active students
       - for each student: load assigned billing categories
         (JSON column on Students or StudentBillingCategories table)
       - for each category: check ApplicableMonths (CSV 1-12),
         check Frequency, calculate discount, skip if already
         invoiced this month
       - if amount > 0, call createInvoice with the school's
         current currency, format invoice number, set dueDate
       - insert Invoice row
       - return summary { schoolId, schoolName, createdCount,
                          invoices: [...] }

18e. PERMISSION CHECK (9-step access check)
================================
Every protected route runs these in order:
  1. requireAuth (src/middleware/auth.js)
       - read JWT from Authorization: Bearer OR kch_token cookie
       - jwt.verify with JWT_SECRET
       - load user from DB
       - check school subscription status (H11)
  2. requireRole(...)              (auth.js or portalAuth.js)
  3. attachSessionContext()        (sessionContextService)
       - build { UserId, ActiveTenantId, ActiveSchoolId, UserRole,
                 DashboardType, IsDevForgeUser, IsSchoolUser,
                 IsParentUser, PermissionSet }
  4. canTenantUseFeature(...)      (entitlementRepository)
       - checks TenantSubscription + SubscriptionPlanFeature +
         TenantFeatureOverride
  5. canTenantUseFeatureWithinLimit(...)  (optional, for high-cost)
  6. requireSchoolPermission(...)   (auth.js)
       - reads req.user.SchoolPermissions from session context
       - checks any-of the listed permissions
  7. service-layer tenant filter
       - every service method requires ActiveTenantId + the record's
         SchoolId / TenantId, throws on mismatch
  8. canUserAccessConversation(...)   (kinderCareHubAccess.js, for KCH)
  9. execute the action
  10. write audit row (audit.js + ConversationAuditLog)

19. ADDING A NEW FEATURE
================================================================================
Pattern (e.g. a new "Consent Forms" feature on the SMS dashboard):

  1. Add a new entity to db/schema.sql
  2. Create src/data/consentRepository.js (parameterised SQL)
  3. Create src/business/consentService.js (business rules)
  4. Create src/application/consentRoutes.js (HTTP routes)
  5. Mount in src/app.js:  app.use('/api/consents', consentRoutes);
  6. Create EJS views:
       src/views/sms/consents/{list, detail, form}.ejs
  7. Add routes to src/application/portal/smsRoutes.js:
       router.get('/consents', ...)
  8. Add sidebar entry to src/views/partials/school-sidebar.ejs
  9. If it needs feature gating, add the SaaSFeature key to db/schema.sql
     (seed) and check via canTenantUseFeature in the route or service
 10. Write tests in tests/consents*.test.js
 11. Run tests/run.js to confirm nothing regressed
