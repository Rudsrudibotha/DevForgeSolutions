2. PER-SCHOOL FLOATING RATE LIMITER (DOS PROTECTION)
================================================================================
NEW FILE:  src/middleware/perSchoolRateLimit.js

Every protected route is now protected by a per-school, per-tenant and
per-user sliding-window rate limiter. The window is 1 minute. Default
limits per minute per school:
   - 600 generic requests
   - 60   message sends
   - 30   image uploads
   - 30   AI chat calls
   - 30   AI reconciliation calls
   - 5    fault-report submissions

The limiter keys are (schoolId, tenantId, userId) in priority order, and
falls back to client IP for unauthenticated traffic. Empty buckets are
reaped every 5 minutes (setInterval(...).unref) to avoid memory leaks.

Routes that have the limiter applied:
   - all /api/messages/*       (KCH messaging)
   - /api/bank-reconciliation (statements list is scoped to bankAccountId)
   - /api/ai/chat
   - /api/ai/reconcile
   - /api/ai/fault-report
   - /api/parent-verification/* (also rate-limited by express-rate-limit)

6. PARENT VERIFICATION (EMAIL + CELLPHONE)
================================================================================
The user asked: "Parents should have access to the parents dashboard
once their email is listed, a verification link should then be sent to
their email to confirm their email (we need to implement a secure way
to sign in here that does not cause difficulty for first time users,
maybe they should enter their cellphone number to confirm that it is
the same)."

NEW FILES:
   src/business/parentVerificationService.js
   src/application/parentVerificationRoutes.js
   src/views/parent/verify.ejs
   src/views/parent/verify-check-email.ejs
   db/schema.sql (added ParentVerificationChallenges +
                  ParentMagicLinks tables)

NEW ENDPOINTS:
   POST /api/parent-verification/start
        Body: { email, cellphone, schoolId? }
        - Verifies the email + cellphone combo against existing parent
          accounts OR ParentRegistrationRequests
        - Sends an email with a 24-hour one-time link
        - Generates a 6-digit SMS code (in dev the code is returned in
          the response so the developer can complete the flow; in
          production the code is sent via SMS gateway and never
          returned to the browser)

   POST /api/parent-verification/complete-email
        Body: { token, email }
        - Validates the one-time link, marks the email as verified
        - Sends the 6-digit SMS code to the email (in production the
          code is sent via SMS; in dev it's also returned)

   POST /api/parent-verification/complete-sms
        Body: { token, email, smsCode }
        - Brute-force protection: max 5 attempts per challenge
        - On success: creates the parent user (if needed), records
          the verification, and emails a 24-hour magic sign-in link
        - The parent has not yet chosen a password; they may use the
          magic link to sign in. Password setting happens on first
          sign-in.

   GET  /api/parent-verification/magic?token=&email=
        - Exchanges the magic link for a real JWT
        - Sets the kch_token HttpOnly cookie for the SSR portal
        - Marks the magic link as used (one-shot)
        - Returns mustSetPassword: true so the portal can prompt
          the parent to choose a password

NEW PUBLIC PAGES (no auth required):
   GET /parent-verify                - the 2-step form (email + cellphone,
                                        then SMS code)
   GET /parent-verify/check-email    - "check your email" landing page

HOW THIS PROTECTS THE PARENT
   - Their email must already be on file with the school (either a
     parent account or a ParentRegistrationRequest). The endpoint
     never creates a parent from a cold email.
   - They must confirm both email and cellphone, which are independent
     channels. An attacker who phishes the email cannot sign in without
     also intercepting the SMS.
   - The 6-digit code is short-lived (10 minutes) and the magic link is
     24 hours, single-use.
   - Rate-limited at 10 challenges per 15 minutes and 20 code attempts
     per 15 minutes.
   - The HTTP-only cookie set by /api/parent-verification/magic is the
     only way the parent gains access. There is no shared password
     guessing surface.

14. RECOMMENDATIONS BEFORE PRODUCTION
================================================================================
1. Set NODE_ENV=production. This enables Strict-Transport-Security,
   forces Secure cookies, hides dev warnings, and disables the
   DISABLE_AUTH test shim.

2. Configure at least one email provider. Without one, the parent
   verification flow returns the SMS code in the response (dev only).
   In production, set:
     EMAIL_PROVIDER=azure
     AZURE_COMMUNICATION_CONNECTION_STRING=endpoint=...
     EMAIL_FROM=Kinder Care Hub <no-reply@yourdomain.com>

3. Configure Hostinger Gemma 3:
     AI_PROVIDER=HOSTINGER_GEMMA3
     AI_BASE_URL=https://your-hostinger-ai-domain.co.za
     AI_API_KEY=<your real key>
   Then run /api/ai/test-connection as a DevForge admin to verify
   the contract.

4. Switch the devforge devforge login from test mode to AAD by
   setting AZURE_AD_TENANT_ID, AZURE_AD_CLIENT_ID,
   AZURE_AD_CLIENT_SECRET, and the ALLOWED_AAD_EMAILS list.

5. Configure JWT_SECRET to a long random value. Rotate periodically.

6. Configure the daily invoice scheduler:
     ENABLE_SCHEDULERS=true

7. Run the database schema migration (db/schema.sql) against the
   Azure SQL production database.

8. Run the test suite (tests/run.js) to confirm the data layer
   tests still pass against the new schema.

15. SAFETY ASSESSMENT
================================================================================
The system is suitable for production use on Azure with the notes
above. Specifically:

   a) Multi-tenant isolation: every query that touches tenant data
      passes through a service that filters by ActiveTenantId. The
      new security upgrades (C1-C4, H5, H6) close the cross-tenant
      IDOR surfaces that existed in the previous legacy build.

   b) The "vanilla model" (no multi-tenancy, one DB per school) is
      not recommended. The SaaS model with strict tenant boundaries
      is more cost-effective and operationally simpler. The audit
      verified that the boundaries are enforced at the service layer
      rather than relying on the client.

   c) Parent data exposure: parents authenticate with their email +
      cellphone, never with a shared password. They see only their
      own children, invoices, and messages. There are no parent-only
      routes that expose school financial controls (bank accounts,
      payroll, audit logs). The portalRoutes mount in app.js does
      not include any /api/* parent routes that touch finance.

   d) DOS protection: per-school + per-user rate limiters are in
      place. The legacy /school shell has been retired (H6) so
      unauthenticated traffic cannot be used as a CSRF pivot.

   e) JWT handling: the JWT is now set in an HttpOnly cookie. Local
      storage still exists for the legacy SPA fallback but is no
      longer the authoritative source.

   f) AI: the AI is read-only, scoped to the caller's tenant, blocked
      for parents and DevForge admins, and never receives data from
      another tenant. Every AI request is logged (sanitised) to
      AIRequestLog.
